# Лабораторна робота №3  
## CQS (Command-Query Separation)

**Мета роботи:**  
Розділити операції запису (команди) та читання (запити) на рівні Application Layer. Студенти
бачать, що читання і запис мають різні вимоги до коду, тестування та оптимізації.

## 1. Що змінилося в структурі проєкту порівняно з лабораторною 2?  

У лабораторній 2 структура проєкту була більш монолітною: основна логіка зосереджувалася у сервісах, які виконували як читання, так і зміну даних. У поточній версії проєкту структура стала більш модульною та чітко розділеною за принципом CQS (Command Query Separation): З’явилися окремі папки для команд (**commands**), запитів (**queries**), їхніх обробників (**commandHandlers, queryHandlers**), а також **DTO для вхідних/вихідних даних**. Це дозволяє чітко розмежувати операції, які змінюють стан системи, і ті, що лише повертають дані.

## 2. Переваги CQS: чим допомогло розділення?

- **Зрозумілість коду:** Кожен клас/метод має одну відповідальність — або змінює стан, або повертає дані.  
- **Тестування:** Unit-тести тепер фокусуються на одному хендлері. Нам не потрібно ініціалізувати величезний сервіс з 10 залежностями, щоб протестувати одну дію.  
- **Масштабованість:** Додаючи нові команди чи запити, не потрібно змінювати існуючі сервіси.  
- **Безпека:** Зменшується ризик випадкової зміни стану при виконанні запитів.  
- **Гнучкість:** Можна легко впроваджувати кешування, логування, валідацію окремо для команд і запитів.  

## 3. Недоліки CQS: що ускладнилося?

- Кількість файлів (Boilerplate): Замість одного файлу сервісу ми отримали 3-5 файлів (на кожну команду/запит + DTO). Це збільшує час на початкове налаштування.
- Більше шаблонного коду: Часто доводиться дублювати схожі структури для різних команд/запитів.
- Початковий поріг входу: Новачкам складніше розібратися у структурі проєкту.

## 4. Чим підхід з Command / Query Handler відрізняється від Service, який робить все сам?

- **Service:** Один клас містить і методи для читання, і для зміни даних. Вся логіка зосереджена в одному місці.  
- **Command/Query Handler:** Для кожної операції створюється окремий обробник. Команди змінюють стан, запити - лише повертають дані. Це дозволяє розділити відповідальності, спростити підтримку та розширення.

## 5. Як CQS впливає на розширюваність? (додати нову команду vs змінити існуючий сервіс)

- **CQS**: Додавання нової команди чи запиту не вимагає змін у вже існуючих обробниках. Це зменшує ризик помилок і конфліктів.
- **Service**: Додавання нової операції часто призводить до зміни існуючого сервісу, що може вплинути на інші частини системи.

## 6. Чи відрізняється структура даних, яку повертає Query, від доменної моделі? Чому це важливо?

Так, структура даних, яку повертає Query, часто відрізняється від доменної моделі. Query може повертати DTO, які оптимізовані для відображення або передачі через API, і не містять зайвої інформації чи поведінки, властивої доменній моделі. Це важливо для:

- **Безпеки**: Не всі поля доменної моделі мають бути видимі зовні.
- **Продуктивності**: DTO можуть містити лише необхідні дані, що зменшує обсяг переданих даних.
- **Гнучкості**: Можна адаптувати структуру відповіді під потреби клієнта, не змінюючи доменну модель.
